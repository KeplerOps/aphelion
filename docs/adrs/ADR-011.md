# ADR-011: Use Java 21 and a JVM-Native Toolchain for the Product Server and Control Layers

- **Status**: Accepted
- **Date**: 2026-04-05
- **Superseded by**: None

## Context

The product layers outside the engine kernel need an implementation language and toolchain. The evaluated paths were full-stack C++, Rust, Go, and Java 21 on the JVM. The kernel remains native code, so the product-layer choice primarily affects boundary overhead, server/control-plane velocity, packaging, testing, and operational ergonomics.

## Decision

Use `Java 21` for the product server and control layers. Keep the engine kernel native. Baseline toolchain: `Gradle`, `JUnit 5`, `Testcontainers`, `jqwik`, `ArchUnit`, `Spotless`, `Error Prone`, and `SpotBugs`. Treat the engine boundary as performance-critical and require explicit performance validation for latency, throughput, concurrency, and memory behavior.

## Consequences

**Positive:**
- Strong enterprise server/tooling ecosystem.
- Good fit for local architecture and testing standards.
- Lower product-layer integration friction while leaving the kernel native.

**Negative:**
- JVM overhead and GC behavior enter the product stack.
- Native boundary remains a core operational concern.
- May require a product-owned bridge if the stock binding is insufficient.

**Risks:**
- Boundary chatter or eager materialization can hurt performance.
- Native failures can still terminate the process if isolation is weak.
- Teams may underinvest in measurement if they assume "native kernel" is enough.
