# ADR-015: Use Layered Conformance and Verification Harnesses

- **Status**: Accepted
- **Date**: 2026-04-05
- **Superseded by**: None

## Context

The system must verify query-language conformance, product language compatibility, internal engine-node behavior, public protocol and driver behavior, durability and recovery, data movement, and performance/resource behavior. openCypher TCK is normative for language conformance, but it is not sufficient for the whole product.

## Decision

Use layered conformance and verification harnesses aligned to explicit compatibility and correctness surfaces. The minimum layers cover language conformance, product language compatibility, internal contract verification, public protocol/driver verification, durability/recovery, data movement/backup, performance/resource validation, and compatibility-surface regression checks. Every named compatibility surface must have executable verification behind it.

## Consequences

**Positive:**
- Compatibility claims are backed by executable verification.
- Reduces drift between language, protocol, storage, and backup behavior and the published contract.
- Gives the team a structured way to validate both correctness and performance.

**Negative:**
- Harness and fixture maintenance cost increases.
- Ownership must stay explicit to avoid duplicated intent.
- Upfront verification work is heavier than an ad hoc strategy.

**Risks:**
- Harness sprawl can blur ownership.
- Teams may add compatibility claims faster than coverage.
- Slow or brittle harnesses can create pressure to bypass them.

## Override Procedure

To deliberately deviate from this ADR:
1. Add `# adr-override: ADR-015` with a rationale comment in the code or plan.
2. Document the deviation and rationale in the PR description.
3. Explain which compatibility or correctness surface lacks executable coverage and why the risk is acceptable.
