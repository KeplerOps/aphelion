# ADR-016: Allocate Aphelion Requirement Ownership Across Kernel, Boundary, and Product Layers

- **Status**: Accepted
- **Date**: 2026-04-06
- **Superseded by**: None

## Context

Aphelion requirements span engine internals, engine-to-product boundary behavior, public server and driver surfaces, control-plane metadata, security, operator tooling, and later clustering concerns. A new kernel repo needs a concrete allocation of ownership so issue placement follows the architecture instead of ad hoc intuition.

The allocation cannot be done by prefix alone. Existing accepted ADRs already constrain ownership:

- [ADR-002](ADR-002.md) keeps the product architecture separate from the engine kernel.
- [ADR-007](ADR-007.md) keeps the public API at the product-server layer.
- [ADR-012](ADR-012.md) uses a co-located out-of-process engine node.
- [ADR-013](ADR-013.md) splits metadata authority between the product and the engine.
- [ADR-014](ADR-014.md) defines a capability-oriented internal engine-node contract.
- [ADR-006](ADR-006.md) makes extension review and packaging product-owned.

A UID-by-UID audit was required before adopting an ownership ADR. That audit is recorded in [ADR-016-requirement-allocation-audit.md](ADR-016-requirement-allocation-audit.md).

## Decision

Use three ownership bands for Aphelion requirements:

- `kernel-owned`
- `boundary-owned`
- `product-owned`

Issue placement shall follow the audited allocation in [ADR-016-requirement-allocation-audit.md](ADR-016-requirement-allocation-audit.md).

Rules:

1. Kernel repo issues come from `kernel-owned` requirements and the kernel implementation side of `boundary-owned` requirements.
2. Aphelion repo issues come from `product-owned` requirements and the product implementation side of `boundary-owned` requirements.
3. Aggregate `*-000` requirements are family umbrellas and are not implementation items by themselves.
4. Any requirement added or materially changed after this ADR that affects ownership must update the audit document before issue placement is treated as settled.
5. When a requirement conflicts with an accepted architectural ADR, the conflict must be resolved by tightening the requirement, revising the ADR, or recording an explicit exception.

## Consequences

**Positive:**
- Gives the new kernel repo a concrete implementation charter.
- Keeps issue placement aligned with the accepted product/kernel boundary.
- Avoids treating entire families such as `APH-CAT`, `APH-API`, or `APH-SEC` as kernel work when the architecture says otherwise.

**Negative:**
- The allocation must be maintained as requirements evolve.
- Split families create paired work across repos and require more coordination.

**Risks:**
- The audit can go stale if requirements change without updating the support document.
- Boundary-owned requirements can still attract ownership drift if issue templates and traceability are sloppy.
- Requirement wording conflicts, especially around extensions and backup/catalog authority, may create churn until cleaned up.

## Enforcement Controls

- [ADR-016-requirement-allocation-audit.md](ADR-016-requirement-allocation-audit.md) -- authoritative UID-by-UID allocation used for issue placement.
- [ADR-002](ADR-002.md), [ADR-007](ADR-007.md), [ADR-012](ADR-012.md), [ADR-013](ADR-013.md), and [ADR-014](ADR-014.md) -- architectural boundary decisions that constrain ownership.
- Ground Control requirement and issue traceability -- links implementation work back to the owning Aphelion requirement.

No dedicated automation enforces this allocation yet. Until that exists, it is enforced through architecture review and traceability discipline.

## Override Procedure

To deliberately deviate from this ADR:
1. Add `# adr-override: ADR-016` with a rationale comment in the code or plan.
2. Document the deviation and rationale in the PR description.
3. Explain which requirement ownership assignment is being changed and why the new allocation is consistent with the accepted architecture.
