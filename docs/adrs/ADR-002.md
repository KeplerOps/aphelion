# ADR-002: Keep the Product Architecture Separate from the Engine Kernel

- **Status**: Accepted
- **Date**: 2026-04-05
- **Superseded by**: None

## Context

If the kernel defines the whole architecture, the product inherits single-node embedded-engine assumptions in places where those aren't acceptable. The main risk is structural dependency on the kernel's boundaries.

## Decision

Product architecture owns system boundaries. Kernel is an internal subsystem behind product-defined interfaces. Minimum boundary: control plane, security/identity layer, metadata/catalog layer, replication/HA layer, engine node interface. Engine node interface covers: query execution, transaction/session lifecycle, schema/index ops, backup/snapshot hooks, health/metrics/profiling, capability/version reporting, extension/feature registration.

## Consequences

**Positive:**
- Engine replacement remains possible without rewriting the product.
- Prevents embedded-engine assumptions from leaking into product layers.
- Strong single-node-first story without being locked to it.

**Negative:**
- Adds interface design and maintenance work.
- Slows initial kernel feature exposure to the product surface.

**Risks:**
- Too thin an interface leaks kernel details.
- Too broad an interface overdesigns for needs that may not materialize.
