# ADR-001: Build capcom as the Product-Owned Engine Kernel

- **Status**: Accepted
- **Date**: 2026-04-06
- **Superseded by**: None

## Context

The target is a commercially viable graph database product that can eventually support enterprise on-prem and cloud-hosted deployments, including high availability, security, reliability, and operational controls.

The team needs an engine kernel that Aphelion can own long-term.

A code audit of Kuzu showed that it remains a strong reference implementation and a useful source of design lessons, but it is archived, C++, and would become an owned fork immediately. The current direction also prioritizes a Rust kernel for safety while keeping Aphelion development moving in parallel.

Evaluated:

- Kuzu as the long-term product kernel
- Kuzu as a temporary reference implementation or stand-in behind the product-owned boundary
- MillenniumDB (GPL v2, not production-ready)
- Graphflow (research artifact)
- a product-owned greenfield kernel

Observed constraints:

- Kuzu is still valuable as a temporary oracle or stand-in because it has a strong modern single-node design and useful implementation lessons.
- MillenniumDB is explicitly not production ready and is licensed under GPL v2, which is a poor default fit for a proprietary commercial derivative.
- Graphflow artifacts are useful implementation references but are not a credible product substrate for an enterprise-grade engine line.
- A product-owned kernel increases implementation risk, but it also aligns with the priority on a Rust-native safety model and avoids long-term dependency on an archived upstream fork.

## Decision

Build `capcom` as the product-owned engine kernel.

Specific commitments:

- The canonical kernel source for Aphelion is the `capcom` repository.
- `capcom` is the long-term execution and storage kernel, not a temporary placeholder.
- `Kuzu` may be used temporarily as a reference implementation, behavior oracle, or stand-in behind the product-owned engine boundary while `capcom` matures.
- `Kuzu` is not the long-term product kernel, compatibility authority, or architectural source of truth.
- The system will not adopt `MillenniumDB` or `Graphflow` as the product kernel base.

This decision is limited to the kernel layer. It does not make a temporary reference engine the product architecture, control plane, or distributed systems foundation.

## Consequences

**Positive:**
- The product owns the kernel direction, safety model, and release control.
- The kernel strategy now aligns with the priority on a Rust-native implementation.
- Kuzu remains available as a temporary oracle or stand-in for parallel development and differential learning.
- The product boundary still preserves the option to replace or remove temporary reference engines later.

**Negative:**
- The team now owns full kernel development rather than incrementally patching an existing fork.
- Schedule risk shifts from fork stewardship to engine implementation.
- Temporary use of a reference engine requires discipline so it does not leak into product contracts.

**Risks:**
- `capcom` may lag product needs if the engine boundary and verification harnesses are weak.
- Dual-track development can accidentally let Kuzu behavior define product semantics.
- Storage, query, and recovery work may take longer than optimistic planning suggests.
