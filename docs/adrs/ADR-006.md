# ADR-006: Control Extensions Through Product-Owned Review and Packaging

- **Status**: Accepted
- **Date**: 2026-04-05
- **Superseded by**: None

## Context

Extensions, plugins, and third-party components create security, operational, and license risks. Kuzu's extension model changed over time. The ecosystem includes MIT, Apache-2.0, and GPL components.

## Decision

Product-owned review and packaging for all extensions. No arbitrary runtime loading. Security, operational, and license review required for every extension. Strong copyleft excluded unless explicitly approved. An allowlist of approved extensions is maintained.

## Consequences

**Positive:**
- Reduces supply-chain risk.
- Predictable runtime behavior -- no surprise extension loading.
- Prevents dependence on abandoned upstream extension infrastructure.

**Negative:**
- Slows third-party extension adoption.
- Adds compliance and review work for each extension.

**Risks:**
- Review process becomes a bottleneck for development velocity.
- Developers bypass controls during local development.

## Enforcement Controls

- `.claude/rules/adr-conformance.md` -- soft guidance for agents.
- `docs/approved-extensions.yaml` -- audit trail of approved extensions.

- `scripts/check-adr-conformance.sh` -- pre-commit hook.
- `.github/workflows/ci.yml` -- `adr-conformance` job in CI.

## Override Procedure

To deliberately deviate from this ADR:
1. Add `# adr-override: ADR-006` with a rationale comment in the code.
2. Document the deviation and rationale in the PR description.
3. Automated hooks will allow the override to pass.
