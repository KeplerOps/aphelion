# ADR-004: Use openCypher as the Initial Language Compatibility Baseline

- **Status**: Accepted
- **Date**: 2026-04-05
- **Superseded by**: None

## Context

Need a concrete language baseline -- public spec, usable by existing graph users, published artifacts.

## Decision

openCypher as initial language compatibility baseline. Query behavior targets openCypher semantics. TCK treated as normative. No multiple query languages in initial product. No broad GQL conformance in phase 1. Product extensions must be explicitly marked.

## Consequences

**Positive:**
- Public spec with ecosystem familiarity.
- Grammar and TCK artifacts are available for validation.

**Negative:**
- Inherits openCypher limitations (spec gaps, areas where behavior is underspecified).

**Risks:**
- Divergence from openCypher semantics creates user confusion.
- TCK alone is not sufficient for full language conformance testing.

## Enforcement Controls

- `.claude/rules/adr-conformance.md` -- soft guidance for agents.
- `docs/language-deviations.yaml` -- audit trail for any deviations from openCypher semantics.
- `tests/architecture/test_opencypher_extensions_marked.pseudo` -- architecture test ensuring extensions are explicitly marked.

## Override Procedure

To deliberately deviate from this ADR:
1. Add `# adr-override: ADR-004` with a rationale comment in the code.
2. Document the deviation and rationale in the PR description.
3. Automated hooks will allow the override to pass.
