# ADR-014: Use a Capability-Oriented Internal Engine-Node Contract

- **Status**: Accepted
- **Date**: 2026-04-05
- **Superseded by**: None

## Context

The product server and the engine node need a product-owned internal contract. A raw kernel-API mirror would let the kernel define the architecture. A text-query-only contract would be too narrow for real database behavior. A plan/AST contract is too invasive and premature for the current phase.

## Decision

Use a capability-oriented internal engine-node contract. Organize it around engine lifecycle, session lifecycle, transaction lifecycle, statement execution, result streaming, control surfaces, and maintenance/recovery hooks. The contract is product-owned, handle-based, streaming-first, and initially text-plus-parameters for query submission. Do not mirror raw kernel classes or make the engine authoritative for DBMS-level control-plane metadata.

## Consequences

**Positive:**
- Preserves the product-owned boundary while covering real DB behavior.
- Fits the out-of-process engine mode and streaming requirements.
- Keeps future kernel replacement realistic.

**Negative:**
- Requires deliberate interface design instead of direct kernel reuse.
- Contract mistakes are more expensive than ad hoc integration mistakes.
- One-off capability leaks will erode the boundary if not controlled.

**Risks:**
- Interface creep can recreate a disguised kernel API.
- Poor granularity can hurt latency and throughput.
- Ownership drift can blur responsibilities around schema, maintenance, and introspection.
