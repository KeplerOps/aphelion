# ADR-013: Split Catalog and Metadata Authority Between the Product and the Engine

- **Status**: Accepted
- **Date**: 2026-04-05
- **Superseded by**: None

## Context

The system requires DBMS-level metadata, security metadata, compatibility metadata, per-database schema metadata, and engine-internal physical/runtime metadata. If the engine owns too much metadata, the DBMS boundary collapses into the kernel. If the product owns too much metadata, schema and index operations become cross-boundary coordination problems.

## Decision

Split authority by metadata class. The product owns DBMS-level catalog, security, compatibility, extension-policy, audit, and future topology metadata. The engine owns per-database schema/index metadata, storage/recovery metadata, and engine-internal transactional/runtime metadata. Each metadata class has exactly one source of truth; projected copies do not become authoritative.

## Consequences

**Positive:**
- Preserves a real product-owned control plane.
- Keeps transactionally local database metadata close to the engine.
- Reduces unnecessary cross-boundary consistency work.

**Negative:**
- Requires explicit metadata classification and discipline.
- Two authoritative catalog domains must coexist cleanly.
- Poorly designed cross-boundary operations become visible quickly.

**Risks:**
- Metadata duplication can drift.
- New metadata classes can become ambiguous without classification.
- Partial failures can leave projected state stale without reconciliation.

## Override Procedure

To deliberately deviate from this ADR:
1. Add `# adr-override: ADR-013` with a rationale comment in the code or plan.
2. Document the deviation and rationale in the PR description.
3. Explain which metadata class changes authority and why duplication or overlap remains safe.
