# ADR-012: Use a Co-Located Out-of-Process Engine Node

- **Status**: Accepted
- **Date**: 2026-04-05
- **Superseded by**: None

## Context

The product server must integrate with a native engine kernel. The evaluated integration modes were in-process embedding, a co-located out-of-process engine node, a separately deployed engine service, and dual embedded/out-of-process modes. The product already commits to a server architecture and a single-node-first product scope.

## Decision

Use a co-located out-of-process engine node. The product server and engine node run as separate processes on the same host and communicate through a product-owned internal interface. Reject in-process embedding as the product architecture, reject remote engine-service deployment for now, and reject dual runtime modes.

## Consequences

**Positive:**
- Real failure, resource, and lifecycle isolation.
- Cleaner restart, diagnostics, and upgrade behavior.
- Better alignment with later HA and control-plane evolution.

**Negative:**
- Adds IPC and lifecycle-management overhead.
- Requires careful contract design for streaming, cancellation, and transactions.
- Small-query latency becomes more sensitive to interface granularity.

**Risks:**
- A chatty internal protocol can become a bottleneck.
- Poor supervision can waste the isolation benefit.
- Weak boundaries can still leak kernel assumptions into the product.
