# ADR-010: Define and Version Compatibility Surfaces From the First External Release

- **Status**: Accepted
- **Date**: 2026-04-05
- **Superseded by**: None

## Context

System evolves across forked kernel, product server, and later distributed capabilities. Without explicit compatibility surfaces, changes break customers unpredictably. Four domains: query language, on-disk storage, backup/export, extensions. A fifth surface (wire protocol) is deferred by ADR-008.

## Decision

Compatibility is an explicit versioned contract from the first external release. Each surface is named and versioned independently. Release notes must state which surfaces changed and how. No single undifferentiated version number for all concerns. Wire protocol surface added when selected.

## Consequences

**Positive:**
- Upgrade discipline from day one.
- Prevents hidden breakage across releases.
- Easier reasoning about HA and replication compatibility when those arrive.

**Negative:**
- Release management overhead increases.
- Requires deliberate documentation effort for every release.

**Risks:**
- Vaguely defined surfaces fail to actually constrain behavior.
- Policy ignored under delivery pressure.
